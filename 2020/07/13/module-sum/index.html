ES6模块和CommonJS模块总结

前言

在做Vue项目的时候，对ES6还不是特别的理解，尤其是一开始对不同模块进行的模块引入，怎么引入这些模块，为什么这样去引入，什么是按需加载模块，这些东西在做项目的时候没有很好的理解，项目做完一段时间的现在，复盘项目时，顺便去复习一下学到的ES6知识，顺便做个小的总结。

语法

ES6 module : import {state, exists, readFile} from 'fs'; ，export{指定输出变量(方法名)} as ReName(重命名输出变量) 使用export default为模块指定默认输出

CommonJS : let { stat, exists, readfile } = require('fs'); 在模块执行时可以用module.exports 定义当前模块对外输出的接口

在ES6中，export规定模版的对外接口，而import用于输入其他模块提供的功能，这里可以使用*做模块的全部引入，但是一般不会这样做，因为一些没有使用过的模块一并引入会导致文件体积过大，影响效率。

ES6在设计的时候思想是尽量的静态化，让资源在编译时就能确定模块的依赖关系和输入输出变量:

这里的所有引入的变量为只读状态，如果有需要的话，是可以对模块进行改写的，但是因为非常难查错，所以我们尽量只对这些资源当成只读状态而不去改变它们。

另外，import无论放置在文件的任何位置，都会有提升至模块头部的效果，跟变量var会有变量提升的效果类似。

为什么ES6会将模块引入设计为尽量的静态化呢，因为静态化模块可以在编译时得到模块中的对象，从而进行静态优化，而不需要等到运行时在能真正获得模块，在运行时的时候是没有办法进行静态优化的，所以这样设计的好处是不少的。

再来说说CommonJS模块化: CommonJS的加载模块是同步的，也就是说他要在加载完成之后才能执行后面的操作，同时CommonJS是可以修改引入变量的值，但是对数据的支持并没有想象中的丰富，对于基本数据类型，他是复制的，也就是说修改引入的基本数据类型不会修改原来模块的值，但是对于复杂数据类型，他是浅拷贝，引用的模块与原先的模块公用同一块内存地址，所以导致修改一处，两处都会发生变化。在require命令第一次加载模块之后，如果需要重新加载，他不会再次执行，而是拿到缓存中的值，以后再加载的时候拿到的也是第一次加载的值，不会发生变化。

对于ES6，对复杂数据类型就有了一定的支持，正是因为之前静态化的设计，所以对一切数据类型，只要执行import命令，就会生成一个只读引用，同时动态的只读引用，在原模块中修改变量值后在引用模块也会发生变化。

区别

  1. CommonJs require() 在node中支持动态加载。
    ES6 import() 在ES2020提案中才支持动态加载。

  2. CommonJS 值拷贝缓存
    ES6 值引用(动态引用)

  3. CommonJS 为运行时加载
    ES6 编译时输出接口

  4. NodeJS加载
    CommonJS 后缀 .cjs, type: commonjs
    ES6 module 后缀 .mjs, type: module

  5. CommonJS require后 顶层this 为当前模块
    ES6 module import后 顶层this 返回undefined，不指向window
