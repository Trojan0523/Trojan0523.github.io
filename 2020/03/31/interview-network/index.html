经典面试题 --网络篇(简写)

计算机网络七层、五层、四层模型各自说说看？(自上而下)

  • OSI七层模型: (1) 应用层 (2)表示层 (3)会话层 (4) 传输层 (5)网络层 (6)数据链路层 (7)物理层

  • HTTP四层模型: (1)应用层 (2) 传输层 (3) 网络层 (4)数据链路层

  • 五层体系结构: (1) 应用层 (2) 传输层 (3) 网络层 (4) 数据链路层 (5) 物理层

一个完整的HTTP请求所包含的七个步骤

  1. 建立HTTP连接
  2. web浏览器向web服务器发送请求行命令
  3. web浏览器发送请求头
  4. 服务端向客户端回送应答
  5. web服务器发送应答头
  6. 服务器端向浏览器发送数据
  7. 服务器端关闭TCP连接

HTTP请求报文和响应报文格式

常用常见首部

  • 通用首部：

    • Date (创建首部时间)
    • Connection (连接管理)
    • cache-control (缓存管理控制)

  • 请求：

    • Host (资源所在服务器)

  • 响应：

    • Location (客户端重新定位到的URI)
    • Server (HTTP服务器的安装信息)

  • 实体：

    • content-type (实体主类类型)
    • content-Encoding (实体主体编码方式)
    • content-language (实体主体自然语言)
    • content-Length (实体主体的字节数)

HTTP 与 HTTPS的区别

  • 1、 HTTP (HyperText transfer protocol): 超文本传输协议。
    2、 HTTPS (HTTP + SSL): 在http的基础上加上(Scurity Socket Layer)安全套接字层。

  • 1、 HTTP以明文方式传输文本内容(没有进行数据加密)。
    2、 HTTPS用SSL证书通过CA认证验证服务器身份，为浏览器和服务器之间进行加密传输，用到非对称密钥加密。

  • 1、 HTTP端口: 80
    2、 HTTPS端口: 443

  • 1、 HTTP连接简单，无状态协议
    2、 HTTPS可以进行加密传输，身份验证。

HTTP状态码

  • 1XX：返回状态信息
    2XX：成功状态码
    3XX：重定向信息
    4XX：客户端错误信息
    5XX：服务端错误信息

  • 常用状态码

    1. 200 OK：请求响应成功
    2. 204 no content：与200相同，但是响应头后没有body数据
    3. 206 Partial Content：部分响应请求成功
    4. 301 永久重定向
    5. 302 资源临时重定向，POST转为GET
    6. 304 Not Modified：资源未作修改，用cache缓存资源
    7. 307 临时重定向 POST发送请求
    8. 400 Bad Request：客户端请求错误，服务端无法处理请求
    9. 401 authorized： 未认证
    10. 403 forbidden：拒绝访问，禁止服务端请求
    11. 404 Not Found： 资源未找到
    12. 500 internal server error：服务端内部错误，一般程序代码出现错误导致
    13. 502 Bad Gateway： 网关错误
    14. 503 server unavailable： 服务器过载，不可用
    15. (补充) 405 Method notAllowed 访问方法不被允许
    16. 414 URI Too Long 客户端请求URI超过服务器允许的范围

HTTP 1.0 和 HTTP 1.1 和HTTP 2.0的区别

    1. HTTP 1.0为短连接,连接无法复用，会出现线头阻塞(Head of Line(HOL) Blocking)
    2. HTTP 1.1默认为持久化连接,一个TCP上可以传送多个HTTP请求和响应，同时增加了Host字段，能实现断点续传，引入管线化(pipeLine)理论，加入缓存处理机制(字段:cache-control)和状态管理等等。
    1. HTTP 2.0解决了线头拥塞的问题，最大的更新是支持多路复用，引入了新的二进制分帧层，不再以文本格式传输数据，支持首部压缩，可以使用服务端进行推送，服务器可以对客户端的一个请求发送多个响应。

什么是HTTP无状态，可以通过什么方式解决

  • HTTP无状态指对事务的处理没有记忆缓存的能力，缺少状态记录

  • 解决方式: 

    1. 第一次访问时生成一个Cookie，二次访问时通过Cookie可以获取其访问状态。
2. 通过Session保存状态信息。
3. 二者区别：Cookie信息存储在本地客户端浏览器中，Session信息则存储在服务器中。

常用的HTTP请求方法

  • get: 用于访问URI，获取资源
  • post: 用于传输信息提交给服务器
  • put: 传输文件
  • head: 获取报文首部
  • delete: 删除文件
  • options: 查询相应URI支持的HTTP方法

Cookie和Session和LocalStorage和SessionStorage的区别

  • Cookie和Session的区别:

    1. Cookie存储在本地浏览器中，Session存储在服务器文件中
2. Session是基于Cookie实现的，具体做法是将SessionID写到Cookie中去，然后发送到服务器

  • Cookie和LocalStorage的区别:

    1. Cookie的大小为4K，而LocalStorage的大小一般为5M到10M不定(不同的浏览器大小不一样)
2. Cookie会被发送到服务器，LocalStorage不会
3. Cookie用来存放用户信息，而LocalStorage用来存放不重要数据

  • LocalStorage和SessionStorage的区别:

    LocalStorage一般不会过期(除非用户自行清除)，SessionStorage退出浏览器(Session断开)就会清除

请你说说TCP和UDP的区别

  • 1、 TCP (Transfer Control Protocol) : 传输控制协议
    2、 UDP (User Datagram Protocol) : 用户数据报协议

  • 1、 TCP提供可靠的服务，会进行一个差错处理，重传机制，是一个尽力而为的服务。
    2、 UDP是不可靠的服务，没有拥塞处理，发送时即使数据丢失也不会进行重传，适合视频通话等应用场景。

  • 1、 TCP为点对点连接
    2、 UDP支持一对多，一对一，多对一和多对多的交互通信。

  • 1、 TCP首部开销为20字节
    2、 UDP首部开销为8字节

  • 1、 TCP逻辑通信通道为全双工可靠信道
    2、 UDP为不可靠信道

说说TCP三次握手和四次挥手,为什么需要三次握手而不是两次或者四次

  • TCP三次握手是用来确认双方是否有接收数据和传输数据的能力，三次握手确保了双方能够进行收发的操作，从而进行连接，少了一次握手会造成一方对其中的一个收发能力的无法确认，多了一次则造成资源的浪费或者连接效率的低下。

  • 三次握手如图所示

  1. 建立连接时，客户端将SYN包标志位至1发送给服务器端，进入监听状态。

  2. 服务器端接收到客户端的SYN包后，将SYN包和ACK包标志位至1，产生随机值seq = k，将SYN和ACK包发送给客户端。

  3. 客户端接收到SYN和ACK包后，检查ACK是否为y+1，ACK标志位是否为1，是则将连接建立

  • 四次挥手如图所示

  • 四次挥手是用于断开连接的，那么断开连接需要确认也就是服务器端是关闭状态，同时客户端也不再传输数据了，所以要有四次挥手的过程

  1. 客户端发送FIN包，用于关闭客户端到服务端的数据传送，此时客户端的状态为:FIN-WAIT-1.

  2. 服务器端收到FIN包之后，发送ACK包给客户端，ACK确认号为收到序列+ 1，此时服务器端状态为: ClOSED-WAIT

  3. 服务器端发送FIN包，用于关闭客户端到服务器端的数据传输，此时服务器端状态为：LAST_ACK

  4. 客户端收到FIN包后，客户端进入TIME-WAIT状态，发送一个ACK给服务器端，确认序号为收到序号+1，服务器端进入ClOSE状态，四次挥手结束

从URL输入到页面展现到底发生什么？

总体来说分为以下几个过程:

  • DNS 解析:将域名解析成 IP 地址
    我们的域名其实是由ip地址进行dns系统解析得到的，所以要让域名找到我们的服务器必须要经过域名的解析翻译成ip地址，再从得到的ip地址去找相应的服务器才能访问，而dns解析首先会在浏览器缓存里面查找相应的DNS记录，如果找不到在去操作系统本地去找，然后再到上一级的路由器缓存里面去找DNS缓存，还是找不到的话继续往上级的ISP(Internet Server Provider)去找对应的DNS服务器，如果还找不到就会向根服务器发出请求进行递归查询。

  • TCP 连接：TCP 三次握手
    在DNS找到相应的ip地址后，浏览器就会向服务器发送HTTP请求，发送请求的同时要建立连接就需要进行TCP三次握手，具体流程请参照上文的三次握手题目。

  • 发送 HTTP 请求
    发送的HTTP请求中也会包括完整的一个请求报文，其中包括请求的报文首部、空行和请求报文的主体内容。请求首部包括请求行，首部实体字段、通用首部字段和请求首部字段，空行用来区分和分割报文首部和报文主体。

  • 服务器处理请求并返回 HTTP 报文
    服务器收到HTTP请求之后会处理发送的请求信息，并且返回后台程序处理的响应结果，后台程序(也就是我们的后端开发，假设为MVC框架模式)会执行一系列的操作，比如将前台返回的数据发送到后端进行数据的处理，返回视图的修改和渲染，或对数据库进行增删查改等操作，或者进行业务逻辑的处理，简而言之就是浏览器首先发送过来的请求先经过控制器进行业务逻辑的处理，然后调用模型，获取数据库的数据并作相应的操作，数据获取后将渲染好的页面和响应信息以响应报文的形式回送给客户端，然后浏览器进行对网页的渲染加载。HTTP响应报文包含响应报文首部、空行和相应报文主体内容，响应报文首部包含状态行(HTTP版本和状态码)、响应首部字段，通用首部字段和实体首部字段。

  • 浏览器解析渲染页面
    浏览器收到回送响应报文后进行解析渲染页面，浏览器解析渲染页面首先会根据HTML解析DOM树，生成DOM节点，根据CSS解析生成CSS规则树，结合DOM树和CSS规则树生成渲染树，根据渲染树计算每一个节点的信息，然后根据计算好的信息去绘制页面。

  • 断开连接：TCP 四次挥手
    参考上文TCP四次挥手，客户端断开与服务器端的连接。
